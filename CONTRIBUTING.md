#  Обов'язкові правила та інструкція для команди (Team 3)

Вітаю в команді! Ми будуємо **Helpdesk System** на сучасному стеку: **Electron + Vue + Tailwind v4**. Щоб ми успішно здали проєкт, отримали максимум балів і не заважали один одному під час розробки, діють наступні правила.

---

### 1.  Структура проєкту (Де писати код)
Кожен працює у своїй ізольованій зоні, щоб уникнути конфліктів у Git:

* **Компоненти UI:** Створюйте власні `.vue` файли виключно в папці `src/renderer/src/components/`.
* **Стилізація:** Використовуйте тільки класи **Tailwind v4**. Не пишіть інлайновий CSS або окремі файли стилів без погодження з Лідом.
* **Типізація:** Обов'язково використовуйте готові інтерфейси з `src/renderer/src/types.ts`. Якщо потрібно змінити структуру об'єкта `Ticket` — спочатку повідомте Ліда.

---

### 2.  Правила роботи з Git (Branching)
**Увага:** Гілка `main` захищена. Прямі пуші в неї заборонені.

1.  **Синхронізація:** Перед початком роботи завжди робіть `git pull origin main`.
2.  **Нова гілка:** Створюйте окрему гілку від актуальної `main`:
    `git checkout -b feature/назва-задачі` (наприклад, `feature/kanban-board`).
3.  **Пуш:** Після завершення задачі відправляйте свою гілку:
    `git push origin feature/назва-задачі`.
4.  **Злиття:** Створюйте **Pull Request (PR)** на GitHub. Код буде перевірено (Code Review) перед мерджем у `main`.

---

### 3.  Стандарти коммітів
Пишіть зрозумілі повідомлення за стандартом **Conventional Commits**:

* `feat: ...` — додавання нового функціоналу.
* `fix: ...` — виправлення помилки/бага.
* `docs: ...` — зміни в документації або коментарях.
* `style: ...` — правки тільки в дизайні (класи Tailwind).

---

###  Швидкий старт для розробника

1.  **Клонування:** `git clone https://github.com/liza-yakovleva/electron-helpdesk-mkr.git`
2.  **Залежності:** `npm install`
3.  **Запуск:** `npm run dev`
4.  **Огляд:** Відкрийте `src/renderer/src/App.vue`, щоб побачити, де підключаються ваші компоненти.

###  SLA та особливості ТЗ №3
Наш проєкт фокусується на **Service Level Agreement (SLA)**:

* **Розробник Канбану:** Має реалізувати візуальну обробку поля `deadline` (індикація прострочених заявок).
* **Розробник логів:** Повинен фіксувати кожну зміну статусу через IPC-канали (звертайтеся до Романа за ТЗ по бекенд-методах).
